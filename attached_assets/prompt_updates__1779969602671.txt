FEATURE 3: RECON SHEET EDIT WITH OTP APPROVAL
==============================================
Allow drivers to request edits to submitted recon sheets. Admin approves via OTP.

Database changes:
- Add to public.recon_sheets:
  * edit_request_status TEXT DEFAULT 'none' (none|pending|approved|rejected)
  * edit_request_reason TEXT
  * edit_request_sent_at TIMESTAMPTZ
  * edit_request_approved_by UUID REFERENCES profiles(id)
  * edit_request_approved_at TIMESTAMPTZ
  * edit_request_rejection_reason TEXT
  
- Create new table public.otp_verifications:
  * id UUID PRIMARY KEY
  * admin_id UUID REFERENCES profiles(id)
  * resource_type TEXT (recon_edit|booking_edit|booking_delete)
  * resource_id UUID
  * otp_code TEXT (6 digits, hashed)
  * created_at TIMESTAMPTZ DEFAULT NOW()
  * expires_at TIMESTAMPTZ (15 minutes from creation)
  * verified_at TIMESTAMPTZ
  * attempts INT DEFAULT 0

Flow:
1. Driver (on driver-dashboard.html) clicks "Request Edit" on submitted recon
2. Modal opens with reason field and list of editable fields (with checkboxes)
3. Driver submits; backend sets edit_request_status='pending'
4. Admin receives email notification with reason and edit list
5. Admin approves by:
   - Receiving email with "Approve Edit Request" button
   - Clicking button opens admin panel
   - System generates 6-digit OTP, sends to admin email
   - Admin enters OTP in modal
   - On successful OTP verification, edit_request_status='approved'
   - Driver gets notification they can now edit
6. Driver edits fields, resubmits
7. Admin can reject request (sends notification with reason)

Implementation files:
- Create supabase/functions/send-otp-email/index.ts
- Create supabase/functions/verify-otp/index.ts
- Extend driver-dashboard.js with recon edit request UI
- Extend admin.js with recon edit approval UI
- Extend app.js with OTP verification helper

FEATURE 6: BOOKING EDIT/DELETE WITH OTP APPROVAL
================================================
Prevent direct editing/deletion. Require OTP verification.

Flow for EDIT:
1. Admin clicks "Edit" on saved booking
2. Form opens with all fields
3. Admin makes changes, clicks "Save"
4. Modal appears: "Changes require OTP verification"
5. System generates OTP, sends to admin email
6. Admin enters OTP
7. On success: booking updated, edit_log entry created
8. On failure (3 attempts): Lock edit for 10 minutes

Flow for DELETE:
1. Admin clicks "Delete" on booking
2. Confirmation modal: "This action requires OTP verification"
3. Admin selects reason from dropdown:
   - "Booking cancelled by client"
   - "Administrative error"
   - "Duplicate booking"
   - "Other (please specify)"
4. Admin submits; OTP email sent
5. Same verification process as edit

Database:
- Add to public.bookings:
  * is_locked BOOLEAN DEFAULT false
  * locked_at TIMESTAMPTZ
  * locked_reason TEXT
  * last_modified_by UUID REFERENCES profiles(id)
  * last_modified_at TIMESTAMPTZ
  * modification_reason TEXT

- Create public.booking_edit_log:
  * id UUID PRIMARY KEY
  * booking_id UUID REFERENCES bookings(id)
  * admin_id UUID REFERENCES profiles(id)
  * action TEXT (edit|delete)
  * reason TEXT
  * old_values JSONB
  * new_values JSONB
  * approved_at TIMESTAMPTZ
  * created_at TIMESTAMPTZ

Implementation:
- admin.js: Intercept save/delete, trigger OTP flow
- Create supabase/functions/approve-booking-change/index.ts
- Store edit history for audit trail
